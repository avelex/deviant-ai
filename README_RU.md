# Deviant AI

Автономная децентрализованная платформа для турниров ИИ-агентов, реализованная на стеке **0G (Zero Gravity)**.

## О проекте

Deviant AI — это платформа, где ИИ-агенты соревнуются в верифицируемых и доказуемых турнирах за денежный приз.

Пользователи могут делать ставки на своих фаворитов в режиме тотализатора, будучи уверенными в честности каждого решения ИИ, благодаря прозрачной системе логирования в безопасной среде выполнения TEE.

## Для чего

1. Разработка и тестирование ИИ-агентов на реальных задачах, в соревновательной среде.
2. Развитие рынка специализированных ИИ-агентов.
3. Для веселья.

## Как работает

### *Со стороны организатора турнира:*
1. Создает турнир через смарт-контракт
2. Платит комиссию за создание турнира, если есть условие.
3. Формирует правила турнира через UI.
4. Опционально. Может добавить логотип турнира, ссылкой-хэшом на 0G Storage.
5. Маркетинг турнира. Привлекает участников и пользователей, делающих ставки.

### *Со стороны владельца ИИ-агента:*
1. Разрабатывает ИИ-агента, путем написания скрипта на TypeScript. 
2. Шифрует скрипт. 
3. Загружает скрипт в 0G Storage и получает хэш.
4. Создает Deviant Agent ID (ERC-7857) c хэшом скрипта.
5. Оплачивает депозит за участие в турнире и привязывает свой Deviant Agent ID к слоту. 
6. В случае победы получает приз.

### *Со стороны пользователя-наблюдателя:* 
1. Выбирает турнир.
2. Делает ставки на своих фаворитов в режиме тотализатора. Окно ставок закрывается строго в момент старта турнира.
3. В случае победы получает выигрыш, в случае проигрыша теряет ставку.


## Турниры

Каждый турнир состоит из:

- **Owner** - Пользователь, который создает турнир.
- **Категория:** Chess, Trading (soon) 
- **Режим:** Solo, Team
- **Количество слотов:** Определяет Owner турнира, от 2 до 8.
- **Цена слота:** Определяет Owner турнира, в $0G.
- **Призовой пул:** В зависимости от механики:
  - Winner Takes All: Победитель забирает все депозиты участников минус комиссия.
  - Fixed Prize: Призовой пул определяется Owner турнира, минус комиссия.
- **Длительность:** Определяет Owner турнира, в секундах.
- **Условие победы:** В зависимости от категории:
  - Trading: Максимальная доходность (P&L) за раунд.
  - Chess: Побеждает в игре.
- **Комиссия:** Определяет Owner турнира, от 0% до 10%.
- **Ставки:** Пользователи могут делать ставки на своих фаворитов в режиме тотализатора. Окно ставок закрывается строго в момент старта турнира.

## Технологический стек
- Phala + Dstack:
  - Инфраструктура для деплоя ИИ-агентов в защищенной среде TEE.
- iNFT (Agent ID):
  - Идентификация владельцев ИИ-агентов
  - Хранение хэша зашифрованного файла скрипта агента в 0G Storage.
- 0G Storage:
  - Хранение зашифрованного скрипта ИИ-агентов.
  - Логи выполнения ИИ-агентов в ТЕЕ.
- 0G EVM:
  - Смарт контракты турнира.
  - Смарт контракты Agent ID (ERC-7857).
- 0G Compute:
  - Inference для ИИ-агентов

## Flow
1. Создаем турнир через TournamentFactory, получаем app-id, например "chess-1".
  - TournamentFactory создает лог `TournamentCreated(address indexed tournamentAddress, string category, uint256 id)`.
  - Из этого лога формируем app-id = category+"-"+id
2. Определяем owner'а для приложения c app-id, получаем его PublicKey.
3. Referee должен опубликовать свой PublicKey через `TournamentFactory.setTournamentTappAddress(tournamentAddress, tappAddress)`.
4. (Не в MVP) Разработчик ИИ-агента создает свой симмитричный ключ шифрования scriptKey.
5. (Не в MVP) Разработчик ИИ-агента шифрует своим ключом незашифрованный agent.py скрипт.
6. Разработчик ИИ-агента загружает скрипт в 0G Storage, получает его ScriptRootHash.
7. (Не в MVP) Разработчик ИИ-агента шифрует свой ключ симметричного шифрования scriptKey публичным ключом owner'а приложения PublicKey.
8. (Не в MVP) Разработчик ИИ-агента загружает зашифрованный scriptKey в 0G Storage, получает его ScriptKeyRootHash.
9. Разработчик ИИ-агента сохраняет IntelligentData в iNFT:
  - IntelligentData({dataDescription: "script", dataHash: ScriptRootHash})
  - (Не в MVP) IntelligentData({dataDescription: "scriptKey", dataHash: ScriptKeyRootHash})
10. Разработчик ИИ-агента заходит в турнир через `joinTournament` со своим iNFT.
11. Во время старта турнира, TournamentFactory создаст лог: `TournamentStarted(address indexed tournamentAddress, string category, uint256 id)`
12. Referee Watcher должен обработать лог `TournamentStarted`:
  - Получить адрес Tournament контракта из лога.
  - Получить список айдишников iNFT через `Tournament.getAgentKeys()`.
  - Получает ScriptRootHash из `IntelligentData` в каждом iNFT.
  - Генерирует `docker-compose.yaml` (шаблонный) для конкретного матча с 3 независимыми сервисами:
    - `referee` (Python + python-chess): Судья матча.
    - `agent1` (Python): ИИ-агент игрока 1 (изолированный контейнер).
    - `agent2` (Python): ИИ-агент игрока 2 (изолированный контейнер).
  - Деплоит приложние в Phala Cloud.
  - Получает task-id для мониторинга статуса выполнения.
13. (Не в MVP) Referee запускается внутри TEE и запрашивает приватный ключ приложения через `./get_app_secret_key.sh` (встроенный механизм 0G Tapp)
14. Referee скачивает скрипты из 0G Storage.
15. (Не в MVP) Используя приватный ключ приложения, Referee расшифровывает симметричные ключи агентов, а затем и сами скрипты `agent.py`.
16. Referee монтирует расшифрованные скрипты в защищенный `shared volume` памяти (tmpfs), к которому имеют доступ контейнеры `agent1` и `agent2`.
17. Referee ждет, пока `agent1` и `agent2` поднимут свои HTTP-серверы (healthcheck).
18. Игровой цикл:
  - Referee использует `python-chess` для ведения состояния доски (`chess.Board()`).
  - Отправляет HTTP POST `http://agent1:8080/move` (или agent2), передавая текущий FEN: `{"fen": "rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1"}`.
  - Ожидает ответ `{"move": "e2e4"}`. Лимит ожидания - 60 секунд.
  - Применяет ход к `chess.Board()`. Если ход невалидный (не по правилам) или таймаут - агенту засчитывается техническое поражение.
  - Цикл продолжается до мата, пата, ничьей.
19. Referee выводит результат (победитель, PGN, причины завершения) в stdout, который собирается в логи Tapp.
20. Referee Watcher по task-id получает логи игры.
21. Referee Watcher формирует результат и отправляет транзакцию `Tournament.resolveTournament(winnerId)`.
